# cloudsysncd Roadmap

这个路线图面向当前的项目定位：`单机服务 + Cloudflare Tunnel + 已配对设备同步`。

目标不是把 `cloudsysncd` 一次性做成一个重量级安全平台，而是先把它打磨成一个稳定、清晰、可开源协作、对普通用户也足够容易启动的工具。

## 当前阶段

当前仓库已经完成了这些基础工作：

- 对外仓库名称统一为 `cloudsysncd`
- 补齐了 `README`、`LICENSE` 和安全审计文档
- 升级了 `tar` 依赖并清理了已知漏洞
- 下载接口改成流式响应，降低了大文件内存放大风险
- 增加了配对后的逐请求 HMAC 鉴权
- 清理了公开提交中的固定公网域名和运行态敏感内容

但从“可以公开”走到“适合长期维护和贡献”，还需要继续补几块能力。

## 设计原则

- 优先提升部署成功率，再做复杂重构
- 优先补设备管理、密钥管理和运行时边界
- 保持本地单机部署简单，不引入不必要的基础设施
- 对协议演进留余地，但不提前过度设计

## 阶段规划

### v1.1 稳定化与发布基础

目标：让项目更像一个标准开源仓库，别人拿到后能稳定跑起来。

重点工作：

- 增加 GitHub Actions，覆盖语法检查、Python 编译检查、依赖审计和最小 smoke test
- 提供 `Dockerfile` 与 `docker-compose.yml`
- 增加 `.env.example` 与更明确的配置说明
- 提供健康检查接口，例如 `/healthz`
- 增加 `SECURITY.md`、`CONTRIBUTING.md` 和 issue / PR 模板
- 把启动流程整理成“3 分钟跑起来”的快速开始

完成标准：

- 新用户可以在本地或 Docker 中快速启动
- 每次提交都能自动跑基础校验
- 项目有明确的漏洞披露和贡献入口

### v1.2 安全与设备管理

目标：把现有协议收紧成一个更可管理的系统。

重点工作：

- 增加设备列表和设备备注名
- 支持撤销单个设备
- 设计并落地密钥轮换流程
- 为 PIN 增加过期时间和基础限速
- 为下载和批量接口增加并发限制、体积限制和更清晰的审计日志
- 把运行时共享目录和源码目录彻底分离

完成标准：

- 可以安全地下线单个已配对设备
- 服务端运行态数据与仓库源码边界清晰
- 配对窗口和下载接口具备基础防滥用能力

### v1.3 易用性与客户端体验

目标：让非作者用户也能顺利安装、配对和同步。

重点工作：

- 为前端增加更明确的状态反馈、进度提示和失败重试
- 将 Python 客户端整理为可安装 CLI
- 增加首次启动向导，自动提示 Cloudflare Tunnel 使用步骤
- 补充一键启动脚本，例如 `scripts/start-with-cloudflare.sh`
- 让 README、部署文档和故障排查文档形成清晰分层

完成标准：

- 用户不需要读源码就能完成部署和配对
- Python 客户端具备更清晰的安装和升级路径
- Cloudflare 场景下的启动说明足够直接

### v1.4 可运维性与升级策略

目标：让项目具备长期维护的可观测性和演进能力。

重点工作：

- 引入结构化日志
- 增加数据备份和恢复说明
- 明确状态文件格式和协议版本
- 设计兼容性策略，避免未来变更直接打断旧客户端
- 为管理接口补充更清晰的错误码和诊断信息

完成标准：

- 服务启动、配对、下载、异常都可追踪
- 版本升级有明确边界和兼容预期
- 维护者可以稳定处理用户反馈和升级问题

### v2.0 可选架构升级

目标：如果未来要把 `cloudsysncd` 做成更强的安全产品，再进入这一阶段。

可能方向：

- 重新设计主密钥生命周期，降低服务端长期持有核心密钥的程度
- 改为按设备独立管理密钥材料
- 引入文件级密钥封装和更细的授权边界
- 设计设备恢复、换机和密钥迁移流程

这一步是大改，不建议在前面几个阶段没有稳定之前提前推进。

## 建议优先级

如果只先做一小批最值的工作，建议优先：

1. GitHub Actions
2. Docker Compose
3. 设备撤销
4. PIN 过期
5. 运行时目录与源码目录分离
6. 首次启动向导

这六项完成后，项目会明显从“作者自己能跑”提升到“其他人也能稳定上手”。

## 不在当前路线图中的事项

以下方向暂时不建议优先推进：

- 多节点集群化
- 复杂权限系统
- 过早拆分微服务
- 大规模插件化架构

这些方向都会增加维护成本，但对当前定位的收益并不高。

## 跟踪方式

路线图建议和 GitHub milestones / issues 对齐：

- `v1.1` 关注发布基础和自动化
- `v1.2` 关注设备管理和安全加固
- `v1.3` 关注上手体验和客户端可用性
- `v1.4` 关注可运维性和协议演进
- `v2.0` 保留为长期可选项
